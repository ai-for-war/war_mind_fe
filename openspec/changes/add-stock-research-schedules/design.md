## Context

The current stock research feature provides a protected `/stocks/research` workspace for manual report creation, report history, report detail rendering, and explicit refresh. The backend now exposes schedule endpoints under `/api/v1/stock-research/schedules` for organization-scoped recurring stock research jobs.

The backend contract supports:
- one `symbol` per schedule
- runtime config with `provider`, `model`, and nullable `reasoning`
- schedule types `every_15_minutes`, `daily`, and `weekly`
- `hour` values interpreted in `Asia/Saigon` for `daily` and `weekly`
- list, detail, create, update, pause, resume, delete, and run-now endpoints

The agreed frontend scope intentionally excludes the `run-now` endpoint because the existing `New Report` flow already covers ad hoc manual reports. The report API also does not currently expose `trigger_type`, `schedule_id`, or `schedule_run_id`, so the UI cannot truthfully show reports generated by a specific schedule.

## Goals / Non-Goals

**Goals:**
- Extend Stock Research into a tabbed workspace with `Reports` and `Schedules`.
- Add complete schedule CRUD and lifecycle controls: create, list, view, edit, pause, resume, and delete.
- Keep schedule forms grounded in the backend schema: symbol, runtime config, type, hour, and weekdays.
- Reuse existing shadcn primitives and existing stock research symbol/runtime patterns.
- Keep errors and mutations consistent with the app convention: `sonner` toasts for request feedback and field-level validation for form issues.
- Preserve `MainLayout` internal-scroll discipline for desktop workspace pages.

**Non-Goals:**
- Expose `run-now` in schedule UI.
- Schedule multiple symbols, watchlists, portfolios, sectors, or batches.
- Add a timezone selector or local timezone conversion for the schedule `hour` field.
- Show per-schedule generated report history until backend report responses expose a stable schedule relationship.
- Add polling, push updates, or automatic refresh for schedules.
- Add new third-party UI packages.

## Decisions

### Decision: Model schedules inside the existing stock research feature area

The implementation will add schedule types, API methods, hooks, and components under `src/features/stock-research/` instead of creating a separate route or top-level feature.

Why:
- Schedules are automation for stock research reports, not a standalone market module.
- The existing page already owns report creation, runtime catalog access, symbol picking, and report history.
- Keeping the code nearby makes it easier to reuse runtime selection and symbol normalization without duplicating business rules.

Alternatives considered:
- Create a separate `/stocks/research/schedules` route.
  Rejected because the user explicitly chose tabs, and a separate route would split a single workflow.
- Put schedules under stock catalog or watchlists.
  Rejected because the backend schedules are scoped by user and organization, not by one catalog table or watchlist.

### Decision: Use shadcn tabs for the page-level mode switch

The page will render `Reports` and `Schedules` using the installed shadcn `Tabs` component, likely with `TabsList variant="line"` to keep the workspace header compact.

Why:
- shadcn docs and local components support the tab composition directly.
- Tabs match the user's requested navigation model.
- A compact tab switch avoids a new sidebar route and keeps the title/action area stable.

Alternatives considered:
- Use manual buttons with local active state.
  Rejected because shadcn already provides accessible tabs and this repo prefers reusable UI primitives.

### Decision: Treat schedule management as snapshot-based with explicit refresh

The schedules tab will fetch schedule list data and selected schedule detail through React Query hooks. A refresh action will refetch schedule list and selected detail, but the page will not poll.

Why:
- The existing stock research workspace is explicitly refresh-driven.
- Schedule list/detail data is operational metadata, not a live execution stream.
- Avoiding polling keeps frontend behavior aligned with current reports UX.

Alternatives considered:
- Poll `next_run_at` or status periodically.
  Rejected because it adds background behavior not needed for this scope and may imply live scheduler monitoring.

### Decision: Use a dedicated schedule dialog with reused symbol/runtime controls

Create and edit flows will use a schedule-specific dialog that reuses the existing symbol picker and extracts reusable runtime-selection logic where practical. The form will use `Field`, `Select`, `ToggleGroup`, and `Dialog`.

Why:
- The schedule body requires `runtime_config` and a nested `schedule` definition, which differs from manual report creation where runtime override can be omitted.
- A dedicated form prevents overloading the existing create-report dialog with automation semantics.
- Extracting pure runtime helpers keeps component files focused on composition and state wiring.

Alternatives considered:
- Extend `StockResearchCreateReportDialog` with schedule mode.
  Rejected because manual queueing and recurring scheduling have different payload rules, labels, validation, and post-submit behavior.

### Decision: Keep status changes as explicit actions, not edit-form fields

Pause and resume will be separate actions, while the edit dialog will focus on symbol, runtime config, and schedule definition.

Why:
- Pause/resume endpoints exist and communicate intent better than patching `status`.
- Keeping status out of the edit form avoids accidental cadence changes plus state changes in one save.
- The UI can show clearer toast feedback for lifecycle transitions.

Alternatives considered:
- Include `status` in the edit dialog.
  Rejected because lifecycle controls are operational actions, not schedule definition fields.

### Decision: Do not expose backend `run-now`

The schedules tab will not show `Run now`, even though the backend supports it.

Why:
- The existing `New Report` action already covers manual report creation.
- Backend `run-now` does not move `next_run_at`, which could confuse users who expect it to affect schedule cadence.
- Removing this action keeps the first implementation simpler and clearer.

Alternatives considered:
- Add a schedule-row `Run now` action.
  Rejected by product direction and because it duplicates manual report creation.

## Risks / Trade-offs

- [Users may expect to inspect reports produced by a schedule] -> Do not fake this relationship; show only schedule metadata until report responses expose `schedule_id` or equivalent.
- [Users may expect local timezone controls] -> Label daily/weekly hour as Vietnam time and avoid a timezone selector because backend treats `hour` as `Asia/Saigon`.
- [Long schedule lists may stretch the route] -> Preserve `max-h-[calc(100dvh-6rem)]`, `min-h-0`, `overflow-hidden`, and internal `ScrollArea` chains.
- [Runtime catalog is unavailable] -> Disable schedule submit when required runtime values cannot be selected and show retry-oriented feedback.
- [Pause/resume/delete mutations leave stale selected detail] -> Invalidate/refetch schedule list and selected detail after successful mutations.
- [Manual report and schedule forms drift] -> Extract shared pure utilities for runtime defaults, selection, formatting, and schedule payload construction where the existing patterns support it.

## Migration Plan

1. Add schedule contract types, normalization helpers, API methods, query keys, and hooks in the stock research feature area.
2. Add the tabbed page composition while preserving the existing reports view as the default tab.
3. Add the schedules workspace components: list/rail, detail panel, create/edit dialog, delete confirmation, loading/empty/error states.
4. Wire create, update, pause, resume, and delete mutations with toast feedback and refetch behavior.
5. Verify route height and internal scrolling in the schedules tab.
6. If rollback is required, remove the schedules tab and schedule data layer without changing the existing reports flow or backend contract.
