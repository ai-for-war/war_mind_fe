# meeting-recorder-runtime Specification

## Purpose
Define the authenticated meeting recorder runtime, browser audio handling, and backend event reduction for live meeting sessions.

## Requirements
### Requirement: Authenticated meeting session startup
The system SHALL provide a frontend meeting recorder runtime that starts only when the browser session is authenticated, the shared Socket.IO transport is connected, an active organization is selected, and both required audio sources are ready. The runtime SHALL create a fresh `stream_id` for each session start attempt and SHALL emit `meeting:start` with the backend-required meeting payload before streaming any audio frames.

The `meeting:start` payload SHALL include:
- `organization_id` from the active organization context
- `stream_id` generated by the frontend for the live meeting stream
- `language` from the user-selected transcription language
- `encoding` set to `linear16`
- `sample_rate` set to `16000`
- `channels` set to `1`

The runtime SHALL allow `title` to be omitted or empty and SHALL not block startup when no title is provided.

#### Scenario: Start session with valid prerequisites
- **WHEN** the user starts a meeting recorder session while authenticated, socket-connected, organization-scoped, and both required audio sources are ready
- **THEN** the runtime emits `meeting:start` with a fresh `stream_id` and the required mono audio contract before any `meeting:audio` event is sent

#### Scenario: Block start when organization context is unavailable
- **WHEN** the user attempts to start the meeting recorder without an active organization context
- **THEN** the runtime does not emit `meeting:start` or `meeting:audio` and marks session startup as failed

#### Scenario: Start without title
- **WHEN** the user starts a meeting recorder session without entering a title
- **THEN** the runtime still emits a valid `meeting:start` payload and does not require title input for startup

### Requirement: Dual-source browser audio capture and mono streaming
The runtime SHALL acquire and manage exactly two audio sources for each session:
- shared meeting tab audio from Chromium tab capture
- local microphone audio from microphone capture

The runtime SHALL validate that each required source exposes an audio track before the session may start. The audio transform pipeline SHALL mix the two captured sources into a single mono stream, resample the output to `16kHz`, encode the output as raw `PCM16`, and emit backend-compatible binary `meeting:audio` frames.

Each emitted `meeting:audio` event SHALL include metadata for the active session, including `stream_id`, `encoding`, `sample_rate`, `channels`, `sequence`, and `timestamp_ms`.

#### Scenario: Both sources available
- **WHEN** tab capture returns an audio track and microphone capture returns an audio track
- **THEN** the runtime marks both sources ready and allows meeting session startup to continue

#### Scenario: Tab capture missing audio
- **WHEN** the user selects a shared tab stream that does not expose an audio track
- **THEN** the runtime rejects session startup and does not open a meeting streaming session

#### Scenario: Stream mono audio after session acceptance
- **WHEN** the backend acknowledges the session with `meeting:started`
- **THEN** the runtime begins emitting binary `meeting:audio` frames whose metadata matches the active `stream_id` and the required `linear16`, `16000`, `1-channel` contract

#### Scenario: One source temporarily silent
- **WHEN** one required source is temporarily silent while the session is active
- **THEN** the runtime continues emitting mono `meeting:audio` frames without changing the negotiated audio contract

### Requirement: Scoped meeting event reduction
The runtime SHALL reduce backend meeting events into client state only for the active organization and meeting session. The runtime SHALL:
- use `meeting:started` to bind the active `stream_id` to the authoritative `meeting_id`
- treat `meeting:final` as realtime draft transcript state keyed by `utterance_id`
- treat `meeting:utterance_closed` as the authoritative transcript timeline keyed by `sequence`
- treat `meeting:note:created` as additive AI note chunks keyed by `meeting_id` and ordered by `from_sequence` and `to_sequence`

The runtime SHALL ignore meeting events whose `organization_id` does not match the active organization context. The runtime SHALL ignore transcript lifecycle events whose `stream_id` or `meeting_id` do not match the active session. For note events that do not include `stream_id`, the runtime SHALL match them by the active `meeting_id`.

#### Scenario: Ignore unrelated organization events
- **WHEN** the client receives a meeting event whose `organization_id` does not match the active organization context
- **THEN** the runtime ignores the event and does not mutate the active meeting session state

#### Scenario: Draft transcript update
- **WHEN** the runtime receives one or more `meeting:final` events for the active session
- **THEN** it updates realtime draft transcript state for the referenced `utterance_id` without committing a canonical transcript entry

#### Scenario: Canonical transcript commit
- **WHEN** the runtime receives `meeting:utterance_closed` for the active meeting
- **THEN** it appends one authoritative closed utterance entry using the event `sequence` and clears any corresponding draft utterance state

#### Scenario: Additive note chunk append
- **WHEN** the runtime receives `meeting:note:created` for the active `meeting_id`
- **THEN** it appends one note chunk ordered by `from_sequence` and `to_sequence` without replacing previously received note chunks

### Requirement: Graceful finalize and forced teardown behavior
The runtime SHALL use `meeting:finalize` as the normal user-initiated stop path for an active session. The runtime SHALL use `meeting:stop` only for forced teardown conditions such as dependency loss, disposal, or local interruption while a live meeting session is still active.

#### Scenario: User stops active meeting
- **WHEN** the user issues a normal stop command while a meeting session is streaming
- **THEN** the runtime emits `meeting:finalize` for the active `stream_id` and transitions the local session into a finalizing state

#### Scenario: Forced teardown during active session
- **WHEN** the runtime tears down an active meeting session because a required dependency is lost or the session is disposed before normal completion
- **THEN** the runtime emits `meeting:stop` for the active `stream_id` when transport is still available and does not attempt graceful finalize

### Requirement: Terminal lifecycle and late note handling
The runtime SHALL treat `meeting:completed`, `meeting:interrupted`, and `meeting:error` as terminal meeting lifecycle signals. On `meeting:completed` or `meeting:interrupted`, the runtime SHALL stop local media capture and audio streaming for the active session, preserve the transcript and note state already received, and continue listening for later `meeting:note:created` events for the same `meeting_id`. On `meeting:error`, the runtime SHALL transition the session into a failed terminal state and stop local streaming work.

The runtime SHALL not attempt to replay or resume the same live meeting session after teardown.

#### Scenario: Completed meeting still receives note tail
- **WHEN** the runtime receives `meeting:completed` for the active session and later receives `meeting:note:created` for the same `meeting_id`
- **THEN** it preserves the completed session state and appends the later note chunk to the active meeting note timeline

#### Scenario: Interrupted meeting still receives note tail
- **WHEN** the runtime receives `meeting:interrupted` for the active session and later receives `meeting:note:created` for the same `meeting_id`
- **THEN** it preserves the interrupted session state and appends the later note chunk to the active meeting note timeline

#### Scenario: Terminal meeting error
- **WHEN** the runtime receives `meeting:error` for the active session
- **THEN** it marks the session as failed, stops local streaming work, and does not attempt to continue the live capture path
