# interview-lab-runtime Specification

## Purpose
Define the authenticated interview runtime, browser media handling, and backend event reduction for interview lab sessions.

## Requirements
### Requirement: Authenticated interview session startup
The system SHALL provide a frontend interview runtime that starts only when the browser session is authenticated and the shared Socket.IO transport is connected. The runtime SHALL create a new `conversation_id` and a new `stream_id` for each session start attempt and SHALL emit `stt:start` with the backend-required interview payload before streaming any audio frames.

The `stt:start` payload SHALL include:
- `conversation_id` generated by the frontend for the interview session
- `stream_id` generated by the frontend for the live STT stream
- `encoding` set to `linear16`
- `sample_rate` set to `16000`
- `channels` set to `2`
- `channel_map` fixed as channel `0 = interviewer` and channel `1 = user`

#### Scenario: Start session with valid prerequisites
- **WHEN** the user starts an interview session while authenticated, socket-connected, and both required audio sources are ready
- **THEN** the runtime emits `stt:start` with a fresh `conversation_id`, a fresh `stream_id`, and the required audio contract before any `stt:audio` event is sent

#### Scenario: Block start when transport is unavailable
- **WHEN** the user attempts to start an interview session while the shared Socket.IO transport is not connected
- **THEN** the runtime does not emit `stt:start` or `stt:audio` and marks the session start as failed

### Requirement: Dual-source browser media capture
The runtime SHALL acquire and manage exactly two audio sources for each session:
- an `interviewer` source from Chromium tab capture
- a `user` source from microphone capture

The runtime SHALL validate that each required source exposes an audio track before the session may start. The runtime SHALL not fall back to mono, mixed audio, or single-source capture for this capability.

#### Scenario: Both sources available
- **WHEN** tab capture returns an audio track and microphone capture returns an audio track
- **THEN** the runtime marks both sources ready and allows session startup to continue

#### Scenario: Tab capture missing audio
- **WHEN** the user selects a tab capture stream that does not expose an audio track
- **THEN** the runtime rejects session startup and does not open a streaming session

#### Scenario: Microphone permission denied
- **WHEN** microphone capture fails or is denied
- **THEN** the runtime rejects session startup and does not open a streaming session

### Requirement: Backend-compatible stereo audio streaming
The runtime SHALL transform the two captured sources into backend-compatible binary audio frames. The audio transform pipeline SHALL:
- normalize each source into one mono speaker lane
- resample output to `16kHz`
- encode output as raw `PCM16`
- interleave the result into `2-channel` binary frames
- map channel `0` to `interviewer`
- map channel `1` to `user`

Each emitted `stt:audio` event SHALL include metadata for the active session, including `stream_id`, `conversation_id`, `encoding`, `sample_rate`, `channels`, `sequence`, and `timestamp_ms`.

#### Scenario: Stream audio after session acceptance
- **WHEN** the backend acknowledges the session with `stt:started`
- **THEN** the runtime begins emitting binary `stt:audio` frames whose metadata matches the active `conversation_id` and `stream_id`

#### Scenario: Silence on one source
- **WHEN** one required source is temporarily silent while the session is active
- **THEN** the runtime continues emitting `2-channel` interleaved audio frames and fills the silent speaker lane with silence samples instead of changing the audio contract

### Requirement: Authoritative transcript turn reduction
The runtime SHALL reduce backend STT events into speaker-specific client state without inventing its own turn-boundary logic. The runtime SHALL treat:
- `stt:partial` as the current preview of an open utterance
- `stt:final` as a newly stabilized fragment within an open utterance
- `stt:utterance_closed` as the authoritative signal that a speaker turn is committed

The runtime SHALL maintain open utterance state separately for `interviewer` and `user`, and SHALL append transcript timeline entries only when `stt:utterance_closed` is received.

#### Scenario: Partial update for interviewer
- **WHEN** the runtime receives `stt:partial` for the `interviewer`
- **THEN** it updates only the interviewer open utterance preview and does not commit a closed transcript entry

#### Scenario: Final fragment before turn close
- **WHEN** the runtime receives one or more `stt:final` events for the same open utterance
- **THEN** it appends those fragments to stable speaker state without treating the utterance as closed

#### Scenario: Utterance close commits timeline
- **WHEN** the runtime receives `stt:utterance_closed` for a speaker
- **THEN** it appends one authoritative closed utterance entry, clears the corresponding open utterance state, and uses the event payload text as the committed turn text

### Requirement: Streamed AI answer aggregation
The runtime SHALL maintain streamed AI answer state keyed by `utterance_id` for interviewer turns. The runtime SHALL:
- create answer state on `interview:answer:started`
- append streamed text on `interview:answer:token`
- complete the answer on `interview:answer:completed`
- treat `interview:answer` as an alias of the final completed answer event
- deduplicate repeated final-answer payloads for the same `utterance_id`

#### Scenario: Token stream builds final answer
- **WHEN** the runtime receives `interview:answer:started`, multiple `interview:answer:token` events, and `interview:answer:completed`
- **THEN** it stores one answer record for the `utterance_id`, appends token text in order, and marks the answer complete with the final text

#### Scenario: Duplicate final answer signals
- **WHEN** the runtime receives both `interview:answer:completed` and `interview:answer` for the same `utterance_id`
- **THEN** it preserves a single completed answer record and does not duplicate the final answer in client state

### Requirement: Hard-stop teardown on transport or source loss
The runtime SHALL treat loss of any required active dependency as a hard-stop condition. Required active dependencies are:
- the shared Socket.IO transport
- the `interviewer` audio source
- the `user` audio source

When any required dependency is lost during an active session, the runtime SHALL stop local audio processing, emit `stt:stop` for the active `stream_id` when transport is still available, and tear down the session without attempting graceful finalize or session resume.

#### Scenario: Interviewer tab sharing ends mid-session
- **WHEN** the interviewer tab audio track ends while the session is streaming
- **THEN** the runtime hard-stops the session, tears down local processing, and does not attempt `stt:finalize`

#### Scenario: Microphone track ends mid-session
- **WHEN** the user microphone track ends while the session is streaming
- **THEN** the runtime hard-stops the session, tears down local processing, and does not attempt `stt:finalize`

#### Scenario: Socket disconnect during active session
- **WHEN** the shared Socket.IO transport disconnects during an active interview session
- **THEN** the runtime tears down the active session locally and does not attempt to resume the same `stream_id`
